## 二、发现的问题

### 问题 1: videoCache 类型不匹配 (严重)

**位置**: [task-runner.ts#L64](file:///d:/code/web/autoOps/src/main/service/task-runner.ts#L64)

```typescript
// TaskRunner 中的 videoCache 是 Map<string, any>
private videoCache = new Map<string, any>()

// 但调用时使用了类型断言
;(this.adapter as any).setVideoCache(this.videoCache)
```

**问题**:
- `DouyinPlatformAdapter.setVideoCache()` 期望类型是 `Map<string, DouyinFeedItem>`
- TaskRunner 传递的是 `Map<string, any>`
- TypeScript 类型不安全

**修复方案**: 使用泛型或接口统一类型

---

### 问题 2: videoCache 在视频切换时可能为空 (严重)

**位置**: [task-runner.ts#L84-100](file:///d:/code/web/autoOps/src/main/service/task-runner.ts#L84-L100)

```typescript
private setupVideoDataListener(): void {
  this.page.on('response', async (response: Response) => {
    // 监听抖音 feed API
    if (url.includes('https://www.douyin.com/aweme/v1/web/tab/feed/')) {
      // 解析并缓存视频数据
      this.videoCache.set(video.aweme_id, video)
    }
  })
}
```

**问题**:
- 视频缓存依赖 API 响应拦截
- 如果用户快速滑动或页面未完全加载，缓存可能为空
- `getCurrentVideoInfo()` 可能返回 null

**修复方案**: 在 goToNextVideo() 后等待缓存更新

---

### 问题 3: 旧版 douyin-task.ts 与 TaskRunner 并存

**位置**: [douyin-task.ts](file:///d:/code/web/autoOps/src/main/service/douyin-task.ts)

这是一个旧的实现，与新的 TaskRunner 架构不一致：
- `DouyinTask` 直接继承 `EventEmitter`
- 有自己的浏览器启动和视频处理逻辑
- 与 `BasePlatformAdapter` 架构不兼容

**问题**:
- 代码冗余
- `ipc/task.ts` 实际使用的是 TaskRunner，不是 DouyinTask

**建议**: 删除或废弃 douyin-task.ts

---

### 问题 4: stop() 函数参数未使用

**位置**: [task.ts (store)](file:///d:/code/web/autoOps/src/renderer/src/stores/task.ts#L132)

```typescript
async function stop() {
  // 函数定义没有参数
  // 但调用时可能传递了 taskId
  addLog('正在停止任务...')
  const result = await window.api.task.stop()
  // ...
}
```

**问题**: 前端 `stopTask()` 调用 `taskStore.stop(selectedTaskId.value)` 但函数不接受参数。

**影响**: 实际上 IPC 的 stop 不需要 taskId（因为全局只有一个 currentTaskRunner），但语义上可能造成困惑。

---

### 问题 5: 事件监听器重复注册

**位置**: [task.ts (store)](file:///d:/code/web/autoOps/src/renderer/src/stores/task.ts#L109-123)

```typescript
window.api.task.onProgress((data) => {
  addLog(data.message)
})

window.api.task.onAction((data) => {
  // ...
})
```

**问题**: 每次调用 `start()` 都会注册新的监听器，但从未移除。
- 如果 start() 被调用多次，监听器会累积
- 日志会重复输出

**修复方案**:
1. 在 start() 开始时移除旧监听器
2. 或使用 once() 而不是 on()

---

### 问题 6: BasePlatformAdapter 缺少 setVideoCache

**位置**: [base.ts](file:///d:/code/web/autoOps/src/main/platform/base.ts)

`BasePlatformAdapter` 没有定义 `setVideoCache()` 和 `getVideoCache()` 方法：
- `DouyinPlatformAdapter` 有自己的实现
- `TaskRunner` 通过 `(adapter as any).setVideoCache()` 调用
- 类型不安全

**修复方案**: 在 BasePlatformAdapter 中添加抽象方法或protected方法

---

### 问题 7: TaskRunner 中的 taskType 判断逻辑问题

**位置**: [task-runner.ts#L312-330](file:///d:/code/web/autoOps/src/main/service/task-runner.ts#L312-L330)

```typescript
if (taskType === 'combo') {
  // combo 类型处理
} else {
  // 非 combo 类型处理
  const operation = settings.operations.find(op => op.type === taskType)
  if (operation && operation.enabled) {
    if (Math.random() <= operation.probability) {
      // 执行操作
    }
  }
}
```

**问题**:
- 如果 taskType 是 'comment' 但 settings.operations 中没有 comment 类型，会跳过
- 如果没有配置 operations 数组，不会执行任何操作

## 三、修复优先级

| 优先级 | 问题 | 影响 |
|--------|------|------|
| P0 | videoCache 类型不匹配 | 可能导致视频信息获取失败 |
| P0 | 视频切换时缓存可能为空 | 任务无法获取视频信息 |
| P1 | 事件监听器重复注册 | 内存泄漏，日志重复 |
| P1 | douyin-task.ts 与新架构不一致 | 代码维护困难 |
| P2 | stop() 参数未使用 | 语义不清晰 |
| P2 | BasePlatformAdapter 缺少 setVideoCache | 类型不安全 |
| P3 | taskType 判断逻辑 | 可能导致操作被跳过 |

---

## 四、建议的修复步骤

### Step 1: 修复 videoCache 类型问题
1. 在 `BasePlatformAdapter` 添加 `setVideoCache()` 和 `getVideoCache()` 方法
2. 使用泛型统一 `Map<string, any>` vs `Map<string, DouyinFeedItem>`

### Step 2: 修复事件监听器重复注册
1. 在 `task.ts (store)` 的 `start()` 开始时清理旧监听器
2. 或使用 `{ once: true }` 选项

### Step 3: 优化视频缓存等待逻辑
1. 在 `goToNextVideo()` 后添加等待
2. 添加重试机制获取视频信息

### Step 4: 清理旧代码
1. 删除或废弃 `douyin-task.ts`
2. 确保所有任务使用 TaskRunner + PlatformAdapter 架构

### Step 5: 统一类型定义
1. 定义 `VideoCache` 接口
2. 在 `TaskRunner` 和各 Adapter 之间共享类型
